# AI Model Status Overview — Design

**Date:** 2026-07-01
**Author:** Niklas Brown
**Status:** Approved (pending spec review)

## Purpose

A single, shareable HTML page that shows which AI model currently offers the best
performance for specific task categories (coding, reasoning & math, image
generation, agentic/tool use). Intended to be hosted on a public GitHub repo
(GitHub Pages) and shared via link inside the company. Benchmark data is
regenerated by a dedicated Claude Code skill whenever a new model is released.

## Goals

- Answer at a glance: "Which model is best for task X right now?"
- Compare major **paid** and **open-source** models side by side.
- Show the cost of using each model: **per-token (API) price** and **subscription
  plan price** (e.g. ChatGPT Plus, Claude Pro).
- Give a direct **link to set up an API key** for each provider.
- Keep data honest: always show *which* benchmark is being measured and link sources.
- Make refreshing the data a one-command skill invocation, with human review before publish.

## Non-Goals

- No live in-browser scraping of third-party sites (fragile, CORS-limited).
- No backend/server; must work as static files on GitHub Pages.
- Not an exhaustive leaderboard — target the top ~10–15 models per category.

## Architecture

Three artifacts:

```
AI Status Overview/
├── index.html      # self-contained page: HTML + CSS + JS inline
├── data.json       # all benchmark data; regenerated by the refresh skill
└── README.md       # how to publish to GitHub Pages + how to refresh
```

Plus a Claude Code skill installed at `~/.claude/skills/ai-benchmark-refresh/`.

**Data/view split:** `index.html` renders whatever is in `data.json`. The skill
only ever rewrites `data.json`. This keeps refreshes safe and reviewable.

**Loading strategy:** `index.html` loads data via `fetch('data.json')` (works on
GitHub Pages / http). Because `fetch` of a local file fails under the `file://`
protocol, the build also inlines the same data as a fallback `<script
id="data-fallback" type="application/json">…</script>` block, so the page renders
when opened by double-click AND when served. `data.json` remains the single
source of truth; the inline fallback is regenerated from it by the skill.

## The Page (`index.html`)

Sections top to bottom:

1. **Header** — title, "last updated" date, one-line explainer of what the page is.
2. **Verdict cards row** — one card per category: "Best for Coding", "Best for
   Reasoning & Math", "Best for Image Generation", "Best for Agentic & Tool Use".
   Each card shows: winner model, its score, the benchmark name, and the runner-up.
3. **Per-category sections** — for each category:
   - Sortable table: model, org, license badge (open vs proprietary), score,
     price in/out ($/M tokens), context window, speed (tokens/sec where relevant).
   - Horizontal bar chart of the top models by the category's primary benchmark.
   - Top score highlighted.
4. **Speed & price view** — comparison of tokens/sec and $/M tokens across models.
5. **Access & Pricing** — one row per provider: token price range, subscription
   plan(s) with monthly price, and a "Get API key" button linking to the
   provider's key-setup page. This is where cost-of-use and onboarding live so the
   category tables stay focused on performance.
6. **Footer** — data sources with links, methodology note, last-updated timestamp.

**Visual design:** produced via the `/frontend-design` skill. Dense "dashboard"
aesthetic; open-source vs proprietary clearly color-coded; readable on desktop and
mobile. No external runtime dependencies bundled that would break offline (charts
drawn with inline SVG or a tiny inline JS charting routine — no CDN requirement).

## Data Schema (`data.json`)

```json
{
  "last_updated": "2026-07-01",
  "sources": [
    { "name": "Artificial Analysis", "url": "https://artificialanalysis.ai" }
  ],
  "providers": [
    {
      "id": "openai",
      "name": "OpenAI",
      "api_key_url": "https://platform.openai.com/api-keys",
      "subscriptions": [
        { "plan": "ChatGPT Plus", "price_month": 20, "currency": "USD" },
        { "plan": "ChatGPT Pro", "price_month": 200, "currency": "USD" }
      ]
    }
  ],
  "categories": [
    {
      "id": "coding",
      "label": "Coding",
      "primary_benchmark": "SWE-bench Verified",
      "unit": "% resolved",
      "higher_is_better": true,
      "models": [
        {
          "model": "Model name",
          "provider_id": "openai",      // links to providers[].id
          "org": "Provider",             // display label
          "license": "open",            // "open" | "proprietary"
          "score": 72.5,
          "price_in": 3.0,               // $ per 1M input tokens (nullable)
          "price_out": 15.0,             // $ per 1M output tokens (nullable)
          "context": 200000,             // tokens (nullable)
          "speed_tps": 90,               // output tokens/sec (nullable)
          "notes": ""
        }
      ]
    }
  ]
}
```

Categories: `coding`, `reasoning-math`, `image-generation`, `agentic-tooluse`.
Each category names its `primary_benchmark` and `unit` so the page states exactly
what is measured. Image-generation entries use an arena Elo score and may leave
price/speed null. Nullable numeric fields render as "—".

Each model references a `provider_id`. The top-level `providers` array holds
provider-wide facts that don't vary per model: the **API-key setup URL** and the
list of **consumer subscription plans** (plan name + monthly price). The Access &
Pricing section is built from `providers`; the per-token cost columns in category
tables come from each model's `price_in` / `price_out`.

## The Refresh Skill (`ai-benchmark-refresh`)

Installed at `~/.claude/skills/ai-benchmark-refresh/SKILL.md`. When invoked, it:

1. Web-searches current top paid + open models per category, drawing from
   Artificial Analysis, LMArena, official model cards, and benchmark leaderboards
   (SWE-bench Verified, LiveCodeBench, GPQA Diamond, AIME, image arena, etc.).
2. Normalizes results into the schema above (top ~10–15 models per category),
   including current API token prices and, per provider, subscription plan prices
   and the API-key setup URL.
3. Presents a diff/summary of proposed changes vs the current `data.json` and
   **asks the user to confirm before writing**.
4. On confirmation, writes `data.json`, updates `last_updated` and `sources`, and
   regenerates the inline fallback block in `index.html`.
5. Reminds the user to commit and push (and mentions the Pages URL).

Trigger: user runs the skill whenever a new notable model is released.

## Initial Data

The first `data.json` will be populated now with real, current benchmark data via
web research, so the page is immediately useful on first publish.

## Success Criteria

- Opening `index.html` (served or via file://) renders verdict cards, all four
  category tables, charts, the Access & Pricing section, and a working footer.
- Access & Pricing shows each provider's token-price range, subscription plan(s)
  with monthly price, and a working "Get API key" link.
- Tables sort by column; top scores highlighted; license badges correct.
- `data.json` validates against the schema; page renders purely from it.
- Running the refresh skill produces an updated `data.json` after user confirmation
  and refreshes the inline fallback, with the page reflecting the new data.

## Risks / Open Questions

- Benchmark comparability: different orgs report differently. Mitigation: name the
  primary benchmark per category and link sources; note caveats in methodology.
- GitHub Pages path: `fetch('data.json')` uses a relative path so it works under a
  project subpath (`user.github.io/repo/`). Verified by the fallback strategy.
